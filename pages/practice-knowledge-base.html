LLM 知识库系统

基于 Karpathy 方法论的个人知识管理，四层架构自动编译。你几乎从不手动编写或编辑 wiki，那是 LLM 的地盘。

核心理念

用 LLM 为各种研究主题建立个人知识库，近期 token 消耗从写代码转向处理知识（Markdown + 图片），最新 LLM 在这件事上表现相当好。

"收集多个来源的原始数据 → LLM 编译成 .md wiki → 通过 CLI 工具做问答并逐步增强 wiki → 全部在 Obsidian 中可视化。你几乎从不手动编写或编辑 wiki，那是 LLM 的地盘。"

四层架构

1⃣ 数据摄取（Data Ingest）

原始数据来源：

  • 文章、论文、仓库、数据集、图片 → raw/ 目录
  • Obsidian Web Clipper（网页→MD）+ 快捷键下载图片
  • 微信文章抓取入库
  • 飞书文档提取

2⃣ Wiki编译（LLM-Compiled Wiki）

内容构成：

  • 摘要、反向链接、概念分类
  • 专题文章、交叉链接
  • wiki/index.md 索引
  • wiki/summary.md 摘要汇总

关键：完全由 LLM 自动维护，人类几乎不碰。

3⃣ 可视化前端（IDE：Obsidian）

Obsidian 作为"前端IDE"，配合插件：

  • Marp（幻灯片）
  • Graph View（知识图谱）
  • Dataview（数据查询）

4⃣ Q&A与输出（Querying & Output）

触发条件：~100篇文章、~40万词

输出形式：

  • Markdown 文档
  • Marp 幻灯片
  • matplotlib 图表
  • 公众号草稿

归档：输出"归档"回Wiki，形成增强循环。

进阶功能

无需复杂RAG

  • LLM 自动维护索引文件和文档摘要
  • ~100篇/40万字规模下，直接让 LLM 读全部内容效果足够好
  • 洞察：并非所有场景都需要 RAG

健康检查（Linting）

  • 发现数据不一致
  • 填补缺失信息（配合网络搜索）
  • 挖掘概念关联，提出新文章候选
  • LLM 主动建议值得探究的问题方向

工作流程

  1. 你给输入 → 我保存到 raw/
  2. 我自动编译 → 更新 wiki/
  3. 你问问题 → 我基于 wiki 回答
  4. 你要求输出 → 我生成公众号草稿并发布

反馈闭环

每次输出后询问：

  1. 要不要把它创建飞书文档？
  2. 对这个结果的满意度怎么样？

根据用户反馈更新 user-preferences.md，影响后续输出风格。
